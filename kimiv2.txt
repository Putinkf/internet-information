## Глава 1: Архитектурная топология JVM-узлов в контексте Fabric-экосистемы

#### 1.1 Инструментарий байт-код-манипуляции и инъекционные паттерны

Fabric Loader реализует механизм трансформации классов через Mixin Framework, использующий ASM (Abstract Syntax Model) для модификации bytecode во время runtime-инициализации. Критически важно понимать, что `MixinBootstrap.init()` инициирует цепочку ClassLoader-ов, создавая изолированные контексты трансформации для каждого модуля (modid).

```java
// Пример инъекции в ServerLoginNetworkHandler для перехвата пакетов
@Mixin(ServerLoginNetworkHandler.class)
public class ServerLoginNetworkHandlerMixin {
    @Inject(method = "onHello", at = @At("HEAD"), cancellable = true)
    private void interceptHandshake(ServerLoginPacketListener.HelloPacket packet, CallbackInfo ci) {
        // Логика верификации в распределенной архитектуре
        if (DistributedAuthManager.getInstance().validateNodeToken(packet.profile())) {
            ci.cancel(); // Прерывание стандартного пайплайна
        }
    }
}
```

**Конфигурационная спецификация mixin:** файл `modid.mixins.json` должен содержать явное указание приоритета (`priority`), минимизирующее конфликты при множественных инъекциях в один и тот же метод. Рекомендуемое значение для критических сетевых компонентов: 1000-1500.

#### 1.2 Netty EventLoopGroup и многопоточная обработка I/O

Minecraft Server на Fabric использует Netty 4.x для асинхронной обработки сетевых соединений. Ключевой параметр `-Dio.netty.eventLoopThreads` определяет размер пула потоков для `EpollEventLoopGroup` (Linux) или `NioEventLoopGroup` (кроссплатформенный). 

**Оптимизационная матрица:**
- Для NUMA-архитектур: количество потоков = (количество физических ядер × 2) + 1
- Для ARM64-кластеров: рекомендуется `KQueueEventLoopGroup` на macOS/BSD
- Отключение `ChannelOption.TCP_NODELAY` недопустимо для latency-critical приложений

```properties
# server.properties - низкоуровневые сетевые тюнинги
network-compression-threshold=256
network-compression-level=6
use-native-transport=true
max-packets-per-tick=100000
```

## Глава 2: Глубокая настройка JVM для high-load окружений

#### 2.1 Сборщики мусора: сравнительный анализ под нагрузкой 1000+ TPS

При работе с распределенными кластерами Minecraft Fabric, где каждый узел обрабатывает >50 000 чанков и поддерживает 500+ конкурентных соединений, выбор GC критичен для предотвращения "stop-the-world" пауз.

**Конфигурация G1GC (для серверов с 16-64GB RAM):**
```bash
java -Xms32G -Xmx32G \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=100 \
  -XX:G1HeapRegionSize=16M \
  -XX:+ParallelRefProcEnabled \
  -XX:G1ReservePercent=20 \
  -XX:InitiatingHeapOccupancyPercent=15 \
  -jar fabric-server-mc.1.20.4-loader.0.15.7-launcher.1.0.0.jar nogui
```

**ZGC для ultra-low-latency (JDK 21+):**
```bash
java -XX:+UseZGC \
  -XX:+ZGenerational \
  -XX:SoftMaxHeapSize=48G \
  -XX:ZCollectionInterval=5 \
  -XX:+AlwaysPreTouch \
  -jar fabric-server.jar
```

#### 2.2 Off-Heap утечки и диагностика Native Memory Tracking

Fabric-модификации часто используют Direct ByteBuffers для передачи данных между native-кодом и JVM. Диагностика:

```bash
-XX:NativeMemoryTracking=summary
jcmd <pid> VM.native_memory summary
```

**Анализ вывода:**
- `Internal` — обычно содержит off-heap буферы Netty
- `Other` — native-вызовы модификаций (LWJGL, JNA)
- `Thread` — стеки потоков (критично при высоком `max-threads`)

## Глава 3: Распределенная архитектура чанков и протокол синхронизации

#### 3.1 SynchronizationContext и многопоточный доступ к Level

Ванильный Minecraft использует полностью синхронизированный доступ к `ServerLevel` через основной серверный поток. Fabric API предоставляет `ServerChunkEvents` для асинхронной обработки, однако любая модификация блок-стейтов должна строго выполняться в контексте `server.execute(() -> {...})`.

**Паттерн Thread-Safe модификации:**
```java
ServerWorld world = server.getWorld(World.OVERWORLD);
ChunkPos targetPos = new ChunkPos(blockPos);

world.getChunkManager().getChunkFutureSyncOnMainThread(targetPos.x, targetPos.z)
    .thenAccept(either -> either.map(
        chunk -> {
            // Безопасная модификация в контексте серверного тика
            server.execute(() -> {
                chunk.setBlockState(blockPos, Blocks.DIAMOND_BLOCK.getDefaultState(), false);
            });
            return null;
        },
        unloaded -> {
            LOGGER.error("Chunk {} not loaded", targetPos);
            return null;
        }
    ));
```

#### 3.2 Custom Payload Packets и бинарная сериализация

Для межсервисной коммуникации в кластере Fabric-серверов рекомендуется использование `ServerPlayNetworking` с кастомными payload-идентификаторами:

```java
public record ClusterSyncPayload(
    UUID nodeId, 
    byte[] compressedChunkData,
    long timestamp,
    int protocolVersion
) implements CustomPayload {
    public static final Id<ClusterSyncPayload> ID = new Id<>(new Identifier("cluster", "sync"));
    
    public static final PacketCodec<RegistryByteBuf, ClusterSyncPayload> CODEC = PacketCodec.tuple(
        Uuids.PACKET_CODEC, ClusterSyncPayload::nodeId,
        PacketCodecs.BYTE_ARRAY, ClusterSyncPayload::compressedChunkData,
        PacketCodecs.VAR_LONG, ClusterSyncPayload::timestamp,
        PacketCodecs.VAR_INT, ClusterSyncPayload::protocolVersion,
        ClusterSyncPayload::new
    );
    
    @Override
    public Id<? extends CustomPayload> getId() { return ID; }
}
```

**Оптимизация сериализации:**
- Использование `VarInt` вместо фиксированных int (экономия 50% трафика для малых значений)
- `Zstd` компрессия для чанков >1KB (уровень 3 для скорости, 9 для сжатия)
- ByteBuf pooling через `PooledByteBufAllocator.DEFAULT` для предотвращения фрагментации heap

## Глава 5: Базы данных и персистентность вне стандартного NBT

#### 5.1 Шардирование playerdata и инвентарей

Стандартное хранение `playerdata/<UUID>.dat` (NBT) становится bottleneck при >1000 активных игроков. Миграция на PostgreSQL с JSONB-колонками:

```sql
CREATE TABLE player_data (
    uuid UUID PRIMARY KEY,
    last_seen TIMESTAMPTZ DEFAULT NOW(),
    inventory JSONB NOT NULL CHECK (jsonb_typeof(inventory) = 'array'),
    ender_chest JSONB,
    position POINT,
    dimension VARCHAR(64),
    data_version INT,
    nbt_hash BYTEA -- для быстрой проверки изменений
);

CREATE INDEX CONCURRENTLY idx_inventory_gin ON player_data USING GIN (inventory jsonb_path_ops);
CREATE INDEX CONCURRENTLY idx_position_gist ON player_data USING GIST (position);

-- Партиционирование по времени последнего входа
CREATE TABLE player_data_2024 PARTITION OF player_data
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```

**Java-реализация адаптера:**
```java
public class PostgresPlayerDataHandler {
    private static final String UPSERT_SQL = """
        INSERT INTO player_data (uuid, inventory, position, dimension)
        VALUES (?, ?::jsonb, point(?,?), ?)
        ON CONFLICT (uuid) DO UPDATE SET
            inventory = EXCLUDED.inventory,
            position = EXCLUDED.position,
            last_seen = NOW()
        """;
    
    public void savePlayer(ServerPlayerEntity player) {
        NbtCompound nbt = new NbtCompound();
        player.writeNbt(nbt);
        
        String jsonInventory = NbtToJsonConverter.convert(nbt.getList("Inventory", 10));
        
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(UPSERT_SQL)) {
            ps.setString(1, player.getUuid().toString());
            ps.setString(2, jsonInventory);
            ps.setDouble(3, player.getX());
            ps.setDouble(4, player.getZ());
            ps.setString(5, player.getWorld().getRegistryKey().getValue().toString());
            ps.executeUpdate();
        } catch (SQLException e) {
            LOGGER.error("Failed to persist player data", e);
        }
    }
}
```

## Глава 7: Безопасность и hardening инфраструктуры

#### 7.1 JWT-аутентификация в кластерной архитектуре

Замена стандартной сессии Mojang на кастомную JWT-схему для межсерверных переходов:

```java
public class JWTAuthHandler {
    private final Algorithm algorithm = Algorithm.HMAC256(System.getenv("CLUSTER_SECRET"));
    
    public String generateClusterToken(GameProfile profile, String targetNode) {
        return JWT.create()
            .withSubject(profile.getId().toString())
            .withClaim("username", profile.getName())
            .withClaim("target_node", targetNode)
            .withClaim("permissions", extractPermissions(profile))
            .withExpiresAt(new Date(System.currentTimeMillis() + 30000)) // 30 сек
            .sign(algorithm);
    }
    
    public DecodedJWT verifyToken(String token) throws JWTVerificationException {
        JWTVerifier verifier = JWT.require(algorithm)
            .withIssuer("fabric-cluster-controller")
            .build();
        return verifier.verify(token);
    }
}
```

#### 7.2 Rate Limiting на уровне Netty

```java
public class RateLimitHandler extends ChannelDuplexHandler {
    private final LoadingCache<InetAddress, RateLimiter> limiters = Caffeine.newBuilder()
        .expireAfterAccess(10, TimeUnit.MINUTES)
        .build(addr -> RateLimiter.create(10.0)); // 10 пакетов/сек
        
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        if (msg instanceof Packet) {
            InetAddress addr = ((InetSocketAddress) ctx.channel().remoteAddress()).getAddress();
            if (!limiters.get(addr).tryAcquire()) {
                ctx.close(); // Принудительное отключение
                LOGGER.warn("Rate limit exceeded for {}", addr);
                return;
            }
        }
        super.channelRead(ctx, msg);
    }
}
```

#### 8.2 Redis как бэкенд для глобального состояния

```java
public class RedisPlayerBridge {
    private final JedisPool jedisPool;
    private static final String PLAYER_PREFIX = "fabric:player:";
    private static final String SERVER_PREFIX = "fabric:server:";
    
    public void handleServerSwitch(UUID playerId, String targetServer) {
        try (Jedis jedis = jedisPool.getResource()) {
            Transaction tx = jedis.multi();
            
            // Удаление из текущего сервера
            tx.srem(SERVER_PREFIX + getCurrentServer(playerId), playerId.toString());
            
            // Добавление в целевой
            tx.sadd(SERVER_PREFIX + targetServer, playerId.toString());
            
            // Обновление метаданных
            tx.hset(PLAYER_PREFIX + playerId, 
                Map.of("currentServer", targetServer,
                       "lastSwitch", String.valueOf(System.currentTimeMillis()),
                       "inventoryHash", calculateInventoryHash(playerId)));
            
            tx.exec();
            
            // Публикация события в канал
            jedis.publish("fabric:switch", 
                playerId + "|" + targetServer + "|" + System.currentTimeMillis());
        }
    }
}
```

## Глава 9: Оптимизация мира и чанков

#### 9.1 Асинхронная генерация через Chunky

Интеграция с плагином Chunky для pre-generation:
```java
@Mixin(ServerChunkManager.class)
public class AsyncPreGenMixin {
    private final ForkJoinPool pregenPool = new ForkJoinPool(
        Runtime.getRuntime().availableProcessors(),
        new WorkerThreadFactory(),
        null, 
        true
    );
    
    public CompletableFuture<Void> pregenerateRegion(ChunkPos center, int radius) {
        List<ChunkPos> chunks = new ArrayList<>();
        for (int x = -radius; x <= radius; x++) {
            for (int z = -radius; z <= radius; z++) {
                chunks.add(new ChunkPos(center.x + x, center.z + z));
            }
        }
        
        return CompletableFuture.allOf(
            chunks.stream()
                .map(pos -> CompletableFuture.runAsync(
                    () -> forceLoadChunk(pos), pregenPool))
                .toArray(CompletableFuture[]::new)
        );
    }
}
```

#### 10.1 Структура пакета Login Start (0x00)

```

--------------------------------------------------------------------------------
Packet ID               VarInt                 0x00
Name                    String (16)            UTF-8, max 16 chars
Has UUID?               Boolean                true/false
UUID (if present)       UUID (128 bits)        Standard UUID format
```

**Wireshark Lua Dissector:**
```lua
minecraft_protocol = Proto("Minecraft", "Minecraft Protocol")

local f_length = ProtoFields.varint("mc.length", "Packet Length", base.DEC)
local f_id = ProtoFields.varint("mc.id", "Packet ID", base.HEX)
local f_data = ProtoFields.bytes("mc.data", "Data")

minecraft_protocol.fields = {f_length, f_id, f_data}

function minecraft_protocol.dissector(buffer, pinfo, tree)
    local length, offset = readVarInt(buffer, 0)
    local subtree = tree:add(minecraft_protocol, buffer(), "Minecraft Protocol Data")
    
    subtree:add(f_length, buffer(0, offset), length)
    
    local packet_id, id_len = readVarInt(buffer, offset)
    subtree:add(f_id, buffer(offset, id_len), packet_id)
    
    -- Диспетчеризация по ID
    if packet_id == 0x00 then
        dissect_login_start(buffer, offset + id_len, subtree)
    end
end
```

#### 10.2 Расшифровка трафика с AES-256-CFB8

После handshake сервер и клиент генерируют shared secret через Diffie-Hellman (1024-bit).

```java
public class MinecraftEncryption {
    public static Cipher createCipher(int mode, SecretKey key) throws Exception {
        Cipher cipher = Cipher.getInstance("AES/CFB8/NoPadding");
        IvParameterSpec iv = new IvParameterSpec(key.getEncoded());
        cipher.init(mode, key, iv);
        return cipher;
    }
    
    public static byte[] decryptPacket(byte[] encrypted, SecretKey key) {
        try {
            Cipher cipher = createCipher(Cipher.DECRYPT_MODE, key);
            return cipher.update(encrypted); // CFB8 позволяет потоковое шифрование
        } catch (Exception e) {
            throw new RuntimeException("Decryption failed", e);
        }
    }
}
```

## Глава 13: Векторизация и SIMD-оптимизации

#### 13.1 Vector API (JEP 448) для обработки чанков

```java
import jdk.incubator.vector.*;

public class VectorizedChunkProcessor {
    private static final VectorSpecies<Integer> SPECIES = IntVector.SPECIES_PREFERRED;
    
    public static void processBlockLight(byte[] lightData, byte[] mask) {
        int i = 0;
        int upperBound = SPECIES.loopBound(lightData.length);
        
        for (; i < upperBound; i += SPECIES.length()) {
            var vLight = ByteVector.fromArray(SPECIES, lightData, i);
            var vMask = ByteVector.fromArray(SPECIES, mask, i);
            
            // Векторизованная операция: lightData[i] = lightData[i] & mask[i]
            var vResult = vLight.and(vMask);
            vResult.intoArray(lightData, i);
        }
        
        // Хвостовая обработка
        for (; i < lightData.length; i++) {
            lightData[i] &= mask[i];
        }
    }
}
```

## Глава 15: Распределенные транзакции и консенсус

#### 15.1 Two-Phase Commit для кросс-серверных операций

Сценарий: передача предмета между серверами в кластере с strong consistency.

```java
public class DistributedTransactionCoordinator {
    private enum TxState { PREPARING, PREPARED, COMMITTING, COMMITTED, ABORTED }
    
    public CompletableFuture<Boolean> transferItem(
        UUID playerId, 
        String fromServer, 
        String toServer,
        ItemStack item
    ) {
        String txId = UUID.randomUUID().toString();
        
        // Phase 1: Prepare
        CompletableFuture<Boolean> prepareFrom = prepareRemove(fromServer, playerId, item, txId);
        CompletableFuture<Boolean> prepareTo = prepareAdd(toServer, playerId, item, txId);
        
        return CompletableFuture.allOf(prepareFrom, prepareTo)
            .thenCompose(v -> {
                if (prepareFrom.join() && prepareTo.join()) {
                    // Phase 2: Commit
                    return CompletableFuture.allOf(
                        commit(fromServer, txId),
                        commit(toServer, txId)
                    ).thenApply(v2 -> true);
                } else {
                    // Rollback
                    rollback(fromServer, txId);
                    rollback(toServer, txId);
                    return CompletableFuture.completedFuture(false);
                }
            });
    }
}
```
